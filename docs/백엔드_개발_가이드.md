## 3단계: API 엔드포인트 구현 (4개)

### API 1. `POST /api/inference` — AI 추론 (키오스크 → 서버)

키오스크가 좌표를 보내면 AI가 분석하고, 대화를 등록하는 가장 핵심 API.

**요청:**

```json
{
  "device_id": "SEOUL_01",
  "keypoints": [0.512, 0.341, 0.001, ..., 0.623]
}
```

- `keypoints`: float 63개 배열 (MediaPipe 랜드마크 21개 x 3좌표)

**처리 흐름:**

```
1. keypoints(63개 float)를 numpy 배열로 변환
2. model.predict() → 인덱스(0~31) 획득
3. model.predict_proba() → 신뢰도 획득
4. label_encoder.inverse_transform() → 한글 글자 변환
   (또는 AI_Label_Map 테이블에서 조회)
5. Communication_Log에 INSERT (status='WAITING')
6. Training_Data_Log에 원본 keypoints JSON 저장
7. 결과 반환
```

**추론 핵심 코드:**

```python
import numpy as np

def inference(keypoints: list[float]):
    x = np.array(keypoints, dtype=np.float32).reshape(1, -1)  # shape (1, 63)
    pred = model.predict(x)[0]                                 # 인덱스
    proba = float(model.predict_proba(x)[0].max())             # 신뢰도
    label = le.inverse_transform([pred])[0]                    # 한글
    return {"label_id": int(pred), "word": label, "confidence": proba}
```

**응답:**

```json
{
  "log_id": 10,
  "recognized_word": "ㅏ",
  "confidence": 0.95
}
```

### API 2. `GET /api/poll/answer` — 답변 확인 (키오스크 폴링)

키오스크가 1초마다 호출하여 역무원 답변이 왔는지 확인.

**요청:**

```
GET /api/poll/answer?device_id=SEOUL_01
```

**처리 흐름:**

```
1. Communication_Log에서 해당 device_id의 가장 최근 레코드 조회
2. status가 COMPLETED이면 답변 반환
3. status가 WAITING이면 대기중 반환
```

**응답 (대기중):**

```json
{
  "status": "WAITING",
  "staff_reply": null
}
```

**응답 (답변 완료):**

```json
{
  "status": "COMPLETED",
  "staff_reply": "개찰구 옆에 있습니다."
}
```

### API 3. `GET /api/staff/list` — 대기 목록 (역무원 화면)

역무원이 답변해야 할 질문 목록 조회.

**요청:**

```
GET /api/staff/list
```

**처리 흐름:**

```
1. Communication_Log에서 status='WAITING'인 레코드 전체 조회
2. 생성시간 오름차순 정렬 (오래된 것부터)
```

**응답:**

```json
[
  {
    "log_id": 10,
    "device_id": "SEOUL_01",
    "recognized_word": "ㅏ",
    "created_at": "2026-04-28T14:30:00"

  {
    "log_id": 11,
    "device_id": "BUSAN_01",
    "recognized_word": "ㅎ",
    "created_at": "2026-04-28T14:31:00"
  }
]
```

### API 4. `POST /api/staff/reply` — 답변 등록 (역무원 → 서버)

역무원이 특정 질문에 답변 작성.

**요청:**

```json
{
  "log_id": 10,
  "reply": "개찰구 옆에 있습니다."
}
```

**처리 흐름:**

```
1. Communication_Log에서 log_id 조회
2. staff_reply 컬럼 업데이트
3. status를 'COMPLETED'로 변경
```

**응답:**

```json
{
  "message": "답변이 등록되었습니다.",
  "log_id": 10
}
```

## 4단계: 전체 데이터 흐름

```
[키오스크 웹캠]
    │
    │  MediaPipe로 손 랜드마크 추출 (프론트에서 처리)
    │  → float 63개 배열
    ▼
POST /api/inference  ← keypoints 전송
    │
    │  ① numpy 변환 → model.predict() → 인덱스
    │  ② label_encoder → 한글 변환
    │  ③ Communication_Log INSERT (WAITING)
    │  ④ Training_Data_Log INSERT (원본 좌표 보관)
    ▼
GET /api/poll/answer  ← 키오스크가 1초마다 폴링
    │
    │  (역무원 답변 대기중...)
    ▼
GET /api/staff/list  ← 역무원 화면에서 대기 목록 확인
    │
    ▼
POST /api/staff/reply  ← 역무원이 답변 작성
    │
    │  Communication_Log UPDATE (COMPLETED)
    ▼
GET /api/poll/answer  ← 키오스크가 다시 폴링 → 답변 수신
    │
    ▼
[키오스크 화면에 답변 표시]
```

## 주의사항

| 항목 | 내용 |
|------|------|
| scikit-learn 버전 | 모델이 `1.7.2`로 학습됨 → `pip install scikit-learn==1.7.2` 필수 |
| 모델 로드 시점 | 서버 시작 시 1회만 로드 (요청마다 로드하면 느림) |
| keypoints 검증 | 배열 길이가 63이 아니면 400 에러 반환 |
| none 처리 | 인식 결과가 `none`(label_id=0)이면 "손 인식 안 됨" 처리 |
| confidence 임계값 | 신뢰도가 너무 낮으면 (예: 0.5 미만) 인식 실패로 처리 권장 |
| CORS 설정 | 키오스크 프론트에서 API 호출하므로 CORS 허용 필요 |
